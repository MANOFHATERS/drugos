# Chemistry (InChIKey conversion)
# RDKit — chemistry toolkit for SMILES to InChIKey conversion
#
# audit-2025 ROOT FIX (issue 37): the previous marker
# ``platform_machine=="x86_64"`` silently skipped rdkit-pypi on ARM64
# hosts (Apple Silicon M1/M2/M3, AWS Graviton EC2, Azure arm64 VMs).
# On those hosts the pipeline's ``convert_to_inchikey()`` degraded to
# returning ``None`` for every SMILES — silently dropping InChIKey
# conversion for the entire run. The fix adds ``platform_machine==
# "aarch64"`` (the Linux ARM64 uname; macOS ARM64 also reports
# "arm64" so we include that too) so RDKit is installed on ARM64
# hosts as well. rdkit-pypi publishes aarch64 wheels since the 2023.9
# release.
rdkit-pypi>=2023.9.1; python_version>="3.8" and (platform_machine=="x86_64" or platform_machine=="aarch64" or platform_machine=="arm64")

# `from airflow.decorators import dag, task` at module top level).

# only by the Docker image), which made every DAG un-importable outside

# validation. Now declared explicitly.
#
# audit-2025 ROOT FIX (issue 38): the previous line had no
# ``python_version`` marker. Apache Airflow 2.x does NOT support
# Python 3.13+ (Airflow 2.10 added 3.12 support; 3.13 support is
# tracked in apache/airflow#39125 and slated for Airflow 3.0).
# Installing this requirements.txt on Python 3.13 would pull in
# Airflow 2.x and then fail at import time with cryptic
# ``AttributeError`` / ``ImportError`` errors. The fix adds the
# ``python_version < "3.13"`` marker so pip skips Airflow on
# Python 3.13+ and prints a clear "no matching distribution" message
# instead. When Airflow 3.0 ships with 3.13 support, bump the version
# constraint and remove the marker.
apache-airflow>=2.8.0; python_version < "3.13"
